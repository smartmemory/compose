# COMP-MOBILE — Mobile PWA companion

**Status:** DESIGN
**Created:** 2026-05-10
**Group:** COMP-MOBILE
**Successor:** COMP-MOBILE-REMOTE (transport + auth, separate ticket)

---

## Why

Compose's cockpit is a desktop-only multi-panel React app — Cytoscape graph, embedded PTY agent terminal, drag-resize layouts, right-click menus. Trying to make every cockpit surface responsive would dilute both experiences. But there's a real mobile use case: **checking on running work and acting on it while away from the desk**. Approving a stuck gate, killing a runaway agent, capturing an idea, glancing at active builds — these don't need the cockpit, they need a phone-shaped surface that talks to the same server.

A separate mobile UX, scoped tightly to those workflows, is honest about the form factor while sharing the data model.

## Goal

A fully functional mobile PWA at `/m` that supports four workflow tracks:

1. **Agents** — list spawned agents (background, multiple) with live status, view their outputs, kill; pending-gate list with approve/revise/kill; send messages to the **single interactive session** when one is active. (Per-spawned-agent chat would need new server APIs and is out of scope; tracked as future `COMP-AGENT-CHAT-PER-ID`.)
2. **Roadmap** — browse items, filter by status/group/keyword, drill into detail, edit `status`/`group`/`confidence`. (Track and tags are not first-class vision-item fields today; deferred. Priority is an ideabox field, not a roadmap field — handled in track 3.)
3. **Ideas** — capture (mobile-natural for shower thoughts), triage with swipe gestures
4. **Builds** — current-build view (reads `active-build.json`, which is last-writer-wins across the runner's per-feature concurrent builds), start a build for a feature (`mode: 'feature' | 'bug'`), watch the log tail, abort. Build history is out of scope (no persistent store today).

Out-of-scope explicitly: graph view (cockpit-only), embedded terminal (cockpit-only), drag-resize panels, right-click menus.

## Non-goals

- **Remote transport.** PWA works on home wifi until `COMP-MOBILE-REMOTE` lands. That ticket adds: bind to 0.0.0.0, a runtime-generated `x-compose-token` (replacing the build-time-baked one for remote sessions), pairing-URL flow, and tunnel guidance. The mobile client is built to be transport-ready (runtime token override in `compose-api.js`) but does not ship the server-side changes here.
- **Native app.** PWA only. App-store / push notifications via real native app is a future ticket if the PWA isn't enough.
- **Tablet-specific layouts.** Phone-first; tablet renders the same single-column shell with breathing room. Distinct two-pane tablet UX is a follow-up if demanded.
- **Mobile cockpit parity.** No graph, no PTY terminal, no design canvas, no docs editor. Mobile is a different surface, not a smaller cockpit.

## Architecture

### Routing

Single Vite app, single `index.html`. `src/main.jsx` reads `window.location.pathname` and renders either the existing `<App/>` (desktop) or `<MobileApp/>` (mobile). No router lib needed; the mobile shell uses `pathname` segments (`/m/<tab>`) and `history.replaceState` for tab persistence — same pattern as the existing popout (`?popout=...`) approach.

```
/             → <App/>          (desktop cockpit)
/popout=...   → <App/>          (popout mode, unchanged)
/m            → <MobileApp/>    (default tab: agents)
/m/agents     → <MobileApp/>    (active tab: agents)
/m/roadmap    → <MobileApp/>
/m/ideas      → <MobileApp/>
/m/builds     → <MobileApp/>
```

### Component tree

```
src/mobile/
  MobileApp.jsx              # root: header, <main>, bottom nav, tab switching
  mobile.css                 # mobile-only styles, no leakage to desktop
  tabs/
    AgentsTab.jsx            # M4 — active agents, gates, chat
    RoadmapTab.jsx           # M2 — list, filter, detail
    IdeasTab.jsx             # M3 — capture, triage
    BuildsTab.jsx            # M5 — start, watch, stop
  components/
    BottomNav.jsx            # 4-button nav (icon + label), aria-pressed
    ItemCard.jsx             # roadmap/ideas card primitive
    AgentCard.jsx            # agent list primitive
    BuildCard.jsx            # current-build primitive (single, from active-build.json)
    StatusPill.jsx           # status indicator (planned/in_progress/blocked/...)
    GatePromptSheet.jsx      # bottom sheet for gate outcome (approve | revise | kill)
    StartBuildSheet.jsx      # bottom sheet form for build start
  hooks/
    useLiveAgents.js         # WS subscription, polled fallback
    useActiveBuild.js
    useRoadmapItems.js       # mirrors useVisionStore but mobile-shaped
    useIdeas.js
```

### Auth (uses existing `x-compose-token`, not Bearer)

The compose server already gates sensitive endpoints (`POST /api/agent/:id/stop`, agent chat, etc.) via `server/security.js:requireSensitiveToken`, which checks an `x-compose-token` request header against `process.env.COMPOSE_API_TOKEN` (auto-generated by `server/supervisor.js` at boot). The desktop cockpit reads the token from `import.meta.env.VITE_COMPOSE_API_TOKEN` at build time and injects it via `src/lib/compose-api.js`.

**Mobile reuses this same path verbatim from M1** — no new auth model. Agent kill, chat, and build control endpoints work because the build-time-baked token rides along on every sensitive request. This means M4/M5 actions are functional on home wifi from day one.

Forward-compatibility for `COMP-MOBILE-REMOTE`:
- `src/lib/compose-api.js` gains a runtime override: `setSensitiveToken(token)` that takes precedence over the build-time value when set
- `MobileApp` reads `?token=…` from the URL once (pairing flow), persists to `localStorage['compose:mobile:sensitiveToken']`, calls `setSensitiveToken`, then strips the param via `replaceState`
- Until `COMP-MOBILE-REMOTE` ships the runtime-generated token + tunnel, the runtime override is unused; the build-time token continues to work
- No `Authorization: Bearer` plumbing — that conflicts with the existing model and isn't needed

### PWA installability

Three additions, behind a `/m`-only check so desktop is unaffected:

1. `public/manifest.webmanifest` — `name`, `short_name`, `start_url: "/m"`, `display: "standalone"`, theme/background color, 192/512 icons.
2. `index.html` — conditional `<link rel="manifest">` + `<meta name="theme-color">` injected when path starts with `/m`. The existing service-worker-killer in index.html stays for desktop dev mode but **becomes conditional**: skip unregister when on `/m` so a registered service worker survives.
3. `public/m-sw.js` — minimal app-shell service worker (cache HTML/JS/CSS for offline boot, network-first for `/api/*`). Registered from `MobileApp` on mount via `navigator.serviceWorker.register('/m-sw.js')`.

PWA registration is **deferred to a polish step at the end of M1**; the shell ships first to validate the layout, manifest+SW second.

### Data flow

Real endpoints, verified against current code:

| Surface | Endpoint | Notes |
|---|---|---|
| Roadmap list | `GET /api/vision/items` | Vision items only — ideas are a separate store |
| Roadmap edit | `PATCH /api/vision/items/:id` | Status/group/confidence editable. **Priority is NOT a vision-item field** — only ideabox has it. Tags also not in the schema. No sensitive-token required for these. |
| Roadmap live | `WebSocket /ws/vision` | Existing reconnect logic in `useVisionStore.js:103` |
| Ideabox list | `GET /api/ideabox` | Returns all ideas |
| Ideabox create / edit | `POST /api/ideabox/ideas`, `PATCH /api/ideabox/ideas/:id` | PATCH explicitly rejects status changes — use the dedicated routes below |
| Ideabox promote / kill / resurrect / discuss / delete | `POST /api/ideabox/ideas/:id/{promote,kill,resurrect,discuss}`, `DELETE /api/ideabox/ideas/:id` | Backed by `docs/product/ideabox.md` |
| Ideabox live | `ideaboxUpdated` WS event | Re-fetch on receipt |
| Agents list | `GET /api/agents/tree` | |
| Agent kill | `POST /api/agent/:id/stop` | **Requires `x-compose-token`** |
| Agent chat / interrupt | per `agent-server.js:122-163` | **Requires `x-compose-token`** |
| Agent stream | `SSE /api/agent/stream` | **Global stream** multiplexed by id; not per-agent. Mobile filters client-side |
| Gates: pending list | `GET /api/vision/gates` | Dedicated route exists |
| Gate detail | `GET /api/vision/gates/:id` | |
| Gate decision | `POST /api/vision/gates/:id/resolve` body `{ outcome: "approve" \| "revise" \| "kill", ... }` | Single resolve route with outcome enum — NOT `/approve` or `/reject` |
| Builds state | `GET /api/build/state` | |
| Build start / abort | **NEW in M5** — `POST /api/build/start`, `POST /api/build/abort` | Wraps existing `runBuild` and `abortBuild` from `lib/build.js`. Keyed by featureCode (no `:id`). Per-feature concurrency: same featureCode active → 409; different featureCode allowed; `active-build.json` is last-writer-wins so the surfaced "current build" reflects the most recent writer. |

**Honest scope:** M2–M4 use only existing endpoints. **M5 ships new server routes** for build start/stop alongside the mobile client work; not a "no new endpoints" feature.

## Phasing

| Phase | Goal | Ships when |
|---|---|---|
| **M1** Shell | `/m` route, mobile.css, bottom nav, tab placeholders, token plumbing, PWA manifest+SW (last) | Phone-friendly skeleton runs, all 4 tabs render placeholders, token round-trip works |
| **M2** Roadmap | List, filter (status/group/keyword), card layout, detail sheet, inline edit (status/group/confidence) | Browse + edit feels native; live updates via existing WS. Track filtering and tags editing deferred (track is description-text scrape on desktop; tags not in schema). |
| **M3** Ideas | Capture form (`POST /api/ideabox/ideas`), list (`GET /api/ideabox`), swipe-to-promote (`POST /api/ideabox/ideas/:id/promote`) / swipe-to-kill (`POST /api/ideabox/ideas/:id/kill`), P0/P1/P2/Untriaged filters | One-thumb capture + triage. Uses ideabox API — NOT vision-items |
| **M4** Agents | Live agent list, status pills, output tail (SSE), kill button, pending gate approval sheet, chat reply | Approve a stuck gate from phone end-to-end |
| **M5** Builds | Current-build view (`active-build.json`), feature picker + mode (`feature`/`bug`), log tail, abort | Start a build from phone, watch progress. Build history out of scope. Concurrency: per-feature; UI surfaces only the most-recent writer of `active-build.json` (matches desktop). |

**Shippability with honest dependencies:**

- **M1** ships standalone — no server work needed beyond what exists.
- **M2** depends only on M1 + existing `/api/vision/items` PATCH.
- **M3** depends only on M1 + existing ideabox routes.
- **M4** depends on M1 + existing `x-compose-token` machinery (already wired in the cockpit at build time, so it works on home wifi from day one). No remote transport required for home-wifi use; remote use waits on `COMP-MOBILE-REMOTE`.
- **M5** depends on M1 **and ships new server endpoints** (`POST /api/build/start`, `POST /api/build/abort`). The build-control HTTP routes don't exist today, but they're thin wrappers over the existing `runBuild` / `abortBuild` functions in `lib/build.js`. M5 is a coupled client+server delivery; no extraction work is required.

M2, M3 can run in parallel after M1. M4 can ship before or after them. M5 is the largest because of the new server endpoints + log-tail UX; sequence it last.

## Design principles

- **One column.** No multi-pane phone layouts. Cards stack. Detail opens as a full-screen overlay or bottom sheet.
- **Bottom nav, not hamburger.** 4 destinations max, each with icon + short label. Hamburgers hide; bottom nav advertises.
- **Touch targets ≥44px.** No tiny buttons.
- **Loading is structural, not a spinner overlay.** Skeleton cards while data hydrates; never a full-screen blocker.
- **No native `prompt`/`confirm`.** Bottom sheets and inline UI; aligned with the same rule the desktop cockpit follows.
- **Optimistic mutations.** Edit status → it changes in the card immediately, server confirms via WS. Rollback on error toast.
- **Offline tolerance.** Service worker caches the shell; `/api/*` requests fail gracefully with a "reconnecting" pill in the header. No queueing of writes in v1.
- **Same WorkspaceProvider, same `wsFetch`.** Mobile is not a parallel data model — it's a different render of the same data the cockpit already has.

## Risks

- **WebSocket on mobile networks.** Cellular networks are flaky; reconnection logic must be solid. Mitigation: reuse the desktop's existing WS reconnect (`useVisionStore.js:103`) and add a visible offline pill.
- **Bundle size.** Mobile shouldn't pull in Cytoscape, Mermaid, KaTeX, Wardley, etc. Vite already does manual chunks but `index.js` is 2.2MB. Mitigation in M1: audit imports — `MobileApp` should not transitively pull desktop-only modules. Likely needs `lazy()` boundaries.
- **PWA + dev-mode SW killer.** index.html unregisters service workers globally. Need to scope that to non-`/m` paths or it kills the PWA install. Already noted above.
- **Token leak via shareable URL.** `?token=…` in the pairing flow is OK because we strip it from the URL via `replaceState` on first load. Document that pairing URLs are single-use and short-lived (covered fully in `COMP-MOBILE-REMOTE`).
- **Theme/dark mode.** Desktop has a theme switcher. Mobile should respect `prefers-color-scheme` by default and offer a toggle in settings. v1: dark-only to match the cockpit; light mode follow-up.
