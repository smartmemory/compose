# Integration Review — STRAT-COMP 4-8 (Plan Phase)

## Status: dirty

## Findings

1. STRAT-COMP-4 and STRAT-COMP-6 define different gate payloads for the same REST contract. `docs/features/STRAT-COMP-4/plan.md:243` and `docs/features/STRAT-COMP-4/design.md:178`/`docs/features/STRAT-COMP-4/design.md:190` make `POST /api/vision/gates` and `GET /api/vision/gates/:id` canonical with `options?` and `comment?`. `docs/features/STRAT-COMP-6/plan.md:57`, `docs/features/STRAT-COMP-6/plan.md:81`, and `docs/features/STRAT-COMP-6/design.md:125`/`docs/features/STRAT-COMP-6/design.md:161` replace that with `artifactAssessment` and omit `options`. Recommended fix: pick one gate schema and update both features before implementation; if `artifactAssessment` stays out of v1, remove it from STRAT-COMP-6 and keep the STRAT-COMP-4 shape.

2. STRAT-COMP-4 and STRAT-COMP-6 disagree on what happens after a successful server probe when gate creation fails. STRAT-COMP-4 requires REST failure to throw with no fallback (`docs/features/STRAT-COMP-4/design.md:148`, `docs/features/STRAT-COMP-4/design.md:161`). STRAT-COMP-6 design matches that fatal behavior (`docs/features/STRAT-COMP-6/design.md:109`/`docs/features/STRAT-COMP-6/design.md:111`), but STRAT-COMP-6 plan still says to log a warning and fall back to readline (`docs/features/STRAT-COMP-6/plan.md:125`). Recommended fix: align STRAT-COMP-6 plan to the STRAT-COMP-4 fatal/no-fallback rule so implementers do not build both branches.

3. STRAT-COMP-4 and STRAT-COMP-6 disagree on gate polling terminal semantics. STRAT-COMP-4 says client-side timeout and `expired` both return `null` and fall back to readline (`docs/features/STRAT-COMP-4/plan.md:283`, `docs/features/STRAT-COMP-4/plan.md:284`, `docs/features/STRAT-COMP-4/design.md:178`). STRAT-COMP-6 design says there is no client-side timeout, `expired` is fatal, and 404 is fatal (`docs/features/STRAT-COMP-6/design.md:168`, `docs/features/STRAT-COMP-6/design.md:170`, `docs/features/STRAT-COMP-6/design.md:179`), while STRAT-COMP-6 plan says `expired` and 404 return `null` (`docs/features/STRAT-COMP-6/plan.md:103`, `docs/features/STRAT-COMP-6/plan.md:106`). Recommended fix: settle one authoritative polling contract across STRAT-COMP-4 and STRAT-COMP-6, then update both STRAT-COMP-6 design and plan to match it.

4. STRAT-COMP-4 and STRAT-COMP-6 do not agree on the canonical API port resolution chain for `probeServer()`. STRAT-COMP-4 requires shared `resolvePort()` with `COMPOSE_PORT > PORT > 3001` and says `lib/server-probe.js` must use it (`docs/features/STRAT-COMP-4/plan.md:31`, `docs/features/STRAT-COMP-4/plan.md:34`, `docs/features/STRAT-COMP-4/design.md:149`). STRAT-COMP-6 plan Task 1 uses only `COMPOSE_PORT` or `3001` (`docs/features/STRAT-COMP-6/plan.md:30`). Recommended fix: make STRAT-COMP-6 explicitly import `resolvePort()` from STRAT-COMP-4 instead of re-specifying probe port logic.

5. STRAT-COMP-4 and STRAT-COMP-7 still have a blocking event-schema authority conflict for `.compose/build-stream.jsonl`. STRAT-COMP-4 declares `event`-keyed records with names like `step_start`, `assistant_text`, and `gate_pending` as canonical (`docs/features/STRAT-COMP-4/design.md:130`, `docs/features/STRAT-COMP-4/design.md:195`, `docs/features/STRAT-COMP-4/design.md:281`). STRAT-COMP-7 explicitly supersedes that with `type`-keyed records and names like `build_step_start`, `assistant`, and `build_gate` (`docs/features/STRAT-COMP-7/design.md:21`, `docs/features/STRAT-COMP-7/design.md:23`, `docs/features/STRAT-COMP-7/plan.md:18`, `docs/features/STRAT-COMP-7/plan.md:63`, `docs/features/STRAT-COMP-7/plan.md:83`). Recommended fix: resolve the source-of-truth conflict before implementation; the chosen schema needs to be reflected in both STRAT-COMP-4 doc-sync tasks and STRAT-COMP-7 writer/bridge tasks.

6. STRAT-COMP-4/5 and STRAT-COMP-8 define incompatible `active-build.json` / `buildState` contracts. STRAT-COMP-4 and STRAT-COMP-5 keep the payload flat and define `violations` as `string[]` (`docs/features/STRAT-COMP-4/design.md:171`, `docs/features/STRAT-COMP-5/design.md:203`, `docs/features/STRAT-COMP-5/design.md:216`). STRAT-COMP-4 also says per-step audit data comes from `.compose/build-stream.jsonl`, not `active-build.json` (`docs/features/STRAT-COMP-4/design.md:217`, `docs/features/STRAT-COMP-4/design.md:282`). STRAT-COMP-8 instead requires `active-build.json` to carry `allStepIds`, `steps[]`, numeric `violations`, and `violationMessages` (`docs/features/STRAT-COMP-8/design.md:260`, `docs/features/STRAT-COMP-8/design.md:264`, `docs/features/STRAT-COMP-8/design.md:266`, `docs/features/STRAT-COMP-8/design.md:347`, `docs/features/STRAT-COMP-8/design.md:355`, `docs/features/STRAT-COMP-8/plan.md:158`). Recommended fix: either move STRAT-COMP-8 back onto the STRAT-COMP-4/7 model (flat `buildState`, per-step history from JSONL) or formally expand the upstream STRAT-COMP-4/5 contract before STRAT-COMP-8 starts.

7. STRAT-COMP-5 and STRAT-COMP-8 disagree on the `GET /api/build/state` response shape. STRAT-COMP-5 defines the endpoint as `{ state: <parsed object> | null }` and consumes it with `data.state ?? null` (`docs/features/STRAT-COMP-5/plan.md:146`, `docs/features/STRAT-COMP-5/plan.md:159`, `docs/features/STRAT-COMP-5/design.md:161`, `docs/features/STRAT-COMP-5/design.md:183`). STRAT-COMP-8 plan expects the body itself to be the build object or `null` and calls `setActiveBuild(data)` (`docs/features/STRAT-COMP-8/plan.md:136`, `docs/features/STRAT-COMP-8/plan.md:137`). Recommended fix: keep STRAT-COMP-5's wrapped response and update STRAT-COMP-8 hydration to unwrap `data.state`.

8. STRAT-COMP-4, STRAT-COMP-6, and STRAT-COMP-8 do not share one canonical step/gate label source. STRAT-COMP-4 says there should be a single shared lookup table in something like `lib/phase-labels.js`, and it explicitly labels `design_gate` as "Design Gate" (`docs/features/STRAT-COMP-4/design.md:241`, `docs/features/STRAT-COMP-4/design.md:247`). STRAT-COMP-6 creates `lib/constants.js` with `GATE_STEP_LABELS` mapping `design_gate` to "Design Review" (`docs/features/STRAT-COMP-6/plan.md:256`, `docs/features/STRAT-COMP-6/plan.md:260`). STRAT-COMP-8 creates a separate `stepLabel()` chain in `build-utils.js` that resolves `design_gate` back to "Design Gate" (`docs/features/STRAT-COMP-8/design.md:440`, `docs/features/STRAT-COMP-8/design.md:444`). Recommended fix: converge on the STRAT-COMP-4 shared-module approach and make both STRAT-COMP-6 and STRAT-COMP-8 import the same table.

9. STRAT-COMP-5, STRAT-COMP-6, and STRAT-COMP-8 have acknowledged temporary overlap on `src/components/StratumPanel.jsx`, but the replacement boundary is not captured as a cross-feature migration contract. STRAT-COMP-5 adds `ActiveBuildBanner` and `FlowList` refresh behavior there (`docs/features/STRAT-COMP-5/plan.md:181`, `docs/features/STRAT-COMP-5/plan.md:194`). STRAT-COMP-6 removes the panel's gate UI in favor of a sidebar link (`docs/features/STRAT-COMP-6/plan.md:241`, `docs/features/STRAT-COMP-6/plan.md:247`). STRAT-COMP-8 then deletes `StratumPanel.jsx` entirely (`docs/features/STRAT-COMP-8/plan.md:263`, `docs/features/STRAT-COMP-8/plan.md:268`). Recommended fix: add an explicit temporary-plumbing note to STRAT-COMP-5 and STRAT-COMP-6 so the team knows those panel changes are transitional and can be minimized once STRAT-COMP-8 starts.

## Summary

The review is dirty. The highest-confidence cross-feature breaks are the gate contract drift between STRAT-COMP-4 and STRAT-COMP-6, the unresolved JSONL event-schema authority conflict between STRAT-COMP-4 and STRAT-COMP-7, and the build-state contract mismatch between STRAT-COMP-4/5 and STRAT-COMP-8. Those three areas would produce incompatible producer/consumer behavior even if each feature were implemented correctly in isolation. I did not find a high-confidence gate-ID or flow-ID format conflict across 4-8, but the shared contracts, event names, and migration boundaries need to be reconciled before implementation proceeds.
